Installation

Step One (Optional) - Install XAMPP

If you know your way around a LAMP stack then feel free to skip this step. This framework relies on Apache and MYSQL and so it is convenient to get your dev environment setup wwith XAMPP.

Download XAMPP

Step Two (Optional) - Setup Virtual Hosts

Setting up a virtual host will allow you to directly access the site from your given address.

Setup Virtual Hosts on Windows

Setup Virtual Hosts on Mac

Setup Virtual Hosts on Linux (Ubuntu)

http://development/

Instead of

http://localhost/simple-mvc-framework/

Step Three - Install Composer

Before you start you must install composer to install the php packages.

You can download composer from here

Run the command in your terminal:

composer install

Step Four - Add .env config

Now you must configure the .env file with appropriate variables

You can find the example.env file in the app directory

<install path>/app/example.env

Add the approritate values (defaulted to XAMPP defaults) and rename the file to

<install path>/app/.env

Step Five - (Optional) Install Phinx globally for database migrations

This step makes database migrations easier

You can install phinx globally with

composer global require robmorgan/phinx

This will allow you to use

phinx <options>

Rather than use

../vendor/bin/phinx <options>

To use phinx you must be in the app directory!

Alternatively you can write a .bashrc script if you know what you are doing.

Step Six - Start Apache and MySQL

Open up XAMPP and click start on Apache and MySQL

Step Seven - Open phpMyAdmin and create a db

Open up XAMPP and click start on the [Admin] button next to MySQL. From here create a new database and make sure the name matches up with the one in the .env file.

Step Eight - Migrate your database with Phinx

This step makes database migrations easier

Change directory to the app folder

cd /app

phinx migrate

Rather than use

../vendor/bin/phinx migrate

To use phinx you must be in the app directory!

Alternatively you can write a .bashrc script if you know what you are doing.

Now you are ready to dev!

Using the framework

Understanding the MVC Model

This framework uses a model view controller model to create it's output. In the simpliest terms we break it down into 3 areas

  • Model - Used for modelling data - basically connects our database to our controller
  • View - Our html templates that render on the page - we use Twig templates
  • Controller - Does the heavy lifting of using models to get the data and renders it out in a view
You can find more information of MVC's here

Understanding routing

Routing in it's simplist terms is taking the url passed into the browser, and routing it to a controller

Behind the scenes a url such as example.com/users/create is converted into example.com/?url=users/create

This ?url=route is then parsed and the approriate controller is called.

Therefore anything passed into the url can be visualised like the following

example.com/<controller>/<method>?<params>
  1. First the params are taken off (if its a GET request)
  2. Whats remaining is split by the forward slashes (/)

  3. It then assumes the method is /index (which is hidden) and looks for a controller

    For example example.com/user/profile would first look for the controller in the directory app/controllers/users/profile.php to see if the index method exists.

    This is because we could be looking for example.com/user/profile/index

  4. If it fails to find the controller for the index method, we then assume the last word is the method name

    For example example.com/user/profile would now look for the controller in the directory app/controllers/users.php to see if the profile method exists.

  5. If the method doesn't exist but the controller does, it will fall back to the index method of that controller.

    For example if the profile method doesn't exist for users, it will basically reroute to example.com/users

  6. If all else fails, if the controller doesn't exist the router will call app/controllers/home.php to call the homepage

Using a controller

Okay so you've read how the router works, and if that's confusing that's okay.

The basic usage of a controller is to name the file and path of the controller to the url you want to route it to

  • example.com/users - becomes app/controllers/users.php calling index()
  • example.com/users/profile - becomes app/controllers/users.php calling profile()
  • example.com/api/roles - becomes app/controllers/api/roles.php calling index()
  • example.com/api/roles/update - becomes app/controllers/api/roles.php calling update()

In the case that there is a conflict between controller/index and controller/method. The controller with the index will take priority

Creating our first route

We are going to make a simple blog page, it won't have much functionality initially but it's to show you how routing works.

First we are going to create a new PHP file in the app/controllersdirectory.

Name the file blog.php

In the file we are going to make a class called Blog extending off Controller
 > app/controllers/blog.php

<?php



class Blog extends Controller

{

    /**

     * Default route for the blog.

     */

    public function index($params)

    {

        echo 'Hello World';

    }

}

Navigating to your site example.com/blog you will now the see the output for "Hello World". Congratulations, you just created your first route!

Using templates

Now controllers wouldn't be very useful if you had to put your html inside of the echo string. Instead what we can do it use templates to render our html, and we can pass data into the template to use.

Lets look the index method again. Since we are extending off of Controller, we have access to a number of methods. The one we're going to look at here is view()

We're going to create a simple template in the views directory. In the app/views directory, create a folder named blog and inside there make a file named index.html

The file path should be app/views/blog/index.html

Inside here create some simple html
> app/views/blog/index.html



<h4>Hello world!</h4>

Inside our controller index method we will now call $this->view(<template path>);

In the file we are going to make a class called Blog extending off Controller
 > app/controllers/blog.php
    
public function index($params)

{

    $this->view('blog/index.html');

}

Now if you navigate to the page again you should now see your rendered template

We can pass data into the template

The view method takes in an array as a second param. It must be a keyed array where the key is the value of the data. Take this example

> app/views/blog/index.html


public function index($params)

{

    $data = array(

        'name' => 'John Smith'

    );



    $this->view('blog/index.html', $data);

}

Now we can access it in our template

> app/views/blog/index.html



<h4>Hello {* name *}!</h4>

Now look at your page and you should now see the name injected into the template!

Things to note

The views here are based on the Twig templating engine

You can find the documentation for it here

You may have notices the syntax {* variable_name *}. This has been modified from the original {{ variable_name }} to prevent conflicts with Vue.js. If you prefer to return to the original syntax you can edit the lexer config in app/core/Controller.php

Checking out the view() method in the Controller, you may notice we have access to variables is_logged_in and route_url, as well as any routes protected by roles for dynamic navbars. Should you need to add variables to every template you can do so here.

Using url params

There are 2 basic types of route. GET and POST. A GET request in simple terms is getting data from an api. Or getting the page to display

We can add params onto a GET route and access them in our controller by adding the synta

?<paramname>=<value>&<paramname>=<value>

For example

example.com/users?id=12

These params are directly accessable in our controller method

> app/views/blog/index.html


public function index($params)

{

    $data = array(

        'name' => $params['name']

    );



    $this->view('blog/index.html', $data);

}

You will want to check that the param you are using exists in the controller before using ofcourse!
